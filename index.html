The business problem

An owner needed three answers on demand — are we profitable, who’s overdue on paying us, and which campaigns are actually worth doing — without rebuilding a spreadsheet formula every time the question changed.

The model design

A six-fact star schema. The key call: InvoiceID is unique in neither the invoice-line fact nor the partial-payments fact, so a Power Query bridge table (one row per invoice) sits between them instead of a broken many-to-many. A single dim_date is marked as the official date table.

The DAX pattern

Open AR / AP are snapshot cards wrapped in CALCULATE(…, ALL(dim_date)) — so a Year slicer can’t silently break a balance whose payments live in an unrelated fact. A TOPN inside ALL() surfaces the top campaign type even under other filters.

The business problem

Leadership tracked KPIs across scattered spreadsheets, so “are we on goal?” took a day to answer and never agreed month to month. They needed one scorecard where every KPI sits next to its YTD target with the trend that explains it.

The model design

A fact table of monthly KPI actuals joins a separate goals table on KPI & period, against a dedicated date table. Keeping targets out of the fact means goals can be re-set without touching a single actual — and the date table keeps the time-intelligence honest.

The DAX pattern

One base measure per KPI, wrapped once in variance logic — DATEADD drives MoM and QoQ, SAMEPERIODLASTYEAR drives YoY. A SWITCH over actual-vs-goal returns the status flag, so every row scores itself from the same tested measures.

Pipeline Orchestration

The n8n workflows that keep the reporting layer fed — scheduled, self-healing extractions from marketing & e-commerce APIs into the tables behind the dashboards above.

Amazon SP-API → Google Sheets Enlarge ↗

Trigger → Auth (OAuth2) → Request async report → ↻ poll until DONE → Download + gunzip → Parse JSON → Split by date → Upsert (keyed on date)

The challenge

Amazon's Sales & Traffic report is generated asynchronously — you request it, and it isn't ready for minutes. A naive “call and read” just fails.

The pattern

Refresh an OAuth2 token, request the report, then poll on a wait → check → conditional-retry loop until it reports DONE. A binary chain follows — download the gzipped document, decompress, parse the JSON — then an idempotent upsert keyed on date so re-runs never duplicate a row.

Shopify Admin API → Google Sheets ×3 Enlarge ↗

Schedule 04:00 → Rolling 5-day window → loop per day → Fetch orders (paginated) → skip empty days → transform ×3 (line-item · order×SKU · totals) → Upsert ×3 → merge → SKU-attributes sub-workflow

The challenge

Shopify orders keep changing after the sale — refunds and status changes land days later. A one-shot “yesterday only” pull would miss them permanently.

The pattern

A rolling 5-day re-pull catches late changes; Link-header pagination handles high-volume days. Each day fans out into three shapes — line-item detail (with a refund-map join), order×SKU, and order-level totals — and idempotent upserts on distinct keys keep the three tables clean. Rate-limit waits and a dedicated error workflow make it unattended-safe.

Staffing Capacity Model

A self-serve Excel model that converts forecasted work volume into the headcount a support team needs each month — and flags where it’s over- or under-staffed. Try it: change the shift length or shrinkage and the projection recomputes live.

Paste your embed code here

Excel for the web → File → Share → Embed

The business problem

Leaders needed to know, each month, how many people the incoming work actually requires — and whether the current team covers it — without re-deriving it by hand every planning cycle.

The model design

Volume (task × month) and per-member worked-days live in structured Tables; a handle-time table maps each task to minutes. Dynamic arrays (UNIQUE, SORT, spill) build the task and roster lists automatically, so adding a task or teammate needs zero formula edits.

The calc pattern

hours = Σ(volume × handle-time) via SUMPRODUCT/XLOOKUP; FTE = ROUNDUP(hours ÷ shift ÷ NETWORKDAYS); HC = FTE × (1 + shrinkage). NETWORKDAYS/EOMONTH make it working-day-aware and respect each member’s hire / offboard dates.

Support KPIs

A clean, self-serve support dashboard built entirely in Google Sheets — ticket volume by product and concern type, plus CSAT. Proof that a sharp, robust summary doesn’t always need a BI license.

Paste your embed code here

Google Sheets → File → Share → Publish to web → Embed

The business problem

Leaders needed one place to see what drives ticket volume — which products, which concerns — and how CSAT tracks, sliceable by period and channel, instead of re-filtering a raw export every time.

The model design

A single flat ticket log (~121K rows) is the one source; four filter cells act as slicers. Every metric is a SUMIFS reading those cells, with IF("All"→"<>") wildcard logic so a blank filter means “all,” wrapped in IFERROR. Share-% is a SUMIFS ratio.

Why just Sheets

No warehouse, no BI license, no infrastructure — a flat log, a few dozen SUMIFS, and clean formatting deliver a dashboard a non-technical team can own and extend themselves. Right-sized beats over-built.

Multi-Source Financial ELT

Six financial systems, six completely different export layouts — indented hierarchies, shifting headers, raw JSON — normalized into one analysis-ready table in Power Query. A drop-a-file-in-the-folder refresh replaced a manual monthly consolidation.

fxSourceA_IncomeStatementPower Query · M
(FileContent as binary) as table =>
// One parser per source system, tuned to that source's quirks.
let
    Source = Excel.Workbook(FileContent, null, true),
    Data = Source{[Name="PL", Kind="Sheet"]}[Data],

    // Pull property name & reporting month as scalars —
    // lighter than temp columns + FillDown on large files
    RawPropertyName = List.First(List.Select(Data[Column1], each _ <> null and Text.StartsWith(_, "Properties:"))),
    PropertyName    = if RawPropertyName <> null then Text.Replace(RawPropertyName, "Properties: ", "") else null,
    RawMonth        = List.First(List.Select(Data[Column1], each _ <> null and Text.StartsWith(_, "As of:"))),
    MonthValue      = if RawMonth <> null then Text.Replace(RawMonth, "As of: ", "") else null,

    // Encode hierarchy depth from the leading whitespace on each row
    AddIndent = Table.AddColumn(Data, "Indent", each
        if [Column1] = null then 0
        else Text.Length([Column1]) - Text.Length(Text.TrimStart([Column1])), type number),

    // Depth 4 = section header (Type); depth 8 = category
    AddType     = Table.AddColumn(AddIndent, "Type", each
        if [Column2] = null and [Column3] = null and [Indent] = 4 then Text.Clean(Text.Trim([Column1])) else null),
    AddCategory = Table.AddColumn(AddType, "Categorization", each
        if [Indent] = 8 then Text.Clean(Text.Trim([Column1])) else null),
    FilledHierarchy = Table.FillDown(AddCategory, {"Type", "Categorization"}),

    // Keep real line items: below category level, not a subtotal, has a value
    FilterDetails = Table.SelectRows(FilledHierarchy, each
        [Indent] >= 8 and
        not Text.StartsWith(Text.Trim([Column1]), "Total") and
        [Column2] <> null and [Column2] <> "Selected Month"),
    RenamedCols = Table.RenameColumns(FilterDetails,
        {{"Column1", "Detail"}, {"Column2", "Value"}}),

    AddProp    = Table.AddColumn(RenamedCols, "Property Name", each PropertyName, type text),
    AddMonth   = Table.AddColumn(AddProp, "Month", each MonthValue, type text),
    FinalTable = Table.SelectColumns(AddMonth,
        {"Detail", "Value", "Month", "Property Name", "Type", "Categorization"})
in
    FinalTable

6 export formats → per-source M parsers → Folder.Files ingestion → union → cleanup → 1 long-format table

The business problem

Six sources each exported monthly financials in a different layout — indented account hierarchies, shifting header rows, merged summary cells, even raw JSON from an API. No single generic parser could handle them, and consolidating by hand every month didn’t scale.

The design

One custom M parser per source absorbs that source’s quirks — dynamic header detection, indent-depth hierarchy, JSON flattening. Folder.Files pulls every file for a source and invokes the matching parser; all six then union into one long-format table, with a final pass stripping subtotals, zeros, duplicates, and noise categories.

The result

A manual monthly consolidation became a refresh-on-open workflow: drop next month’s file in the folder and it flows through to the same clean schema, ready for Power Pivot and the dashboard — no per-file rework.
